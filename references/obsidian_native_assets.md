# Obsidian Native Assets

Use these conventions when the user wants architecture records to be useful inside
Obsidian, not just stored as plain Markdown.

## Markdown notes

- Use frontmatter properties already generated by the scripts: `type`, `project`,
  `status`, `created`, `updated`, `tags`, `source_repo`, and `related_files`.
- ADR notes also use `decision_id`, `decision_scope`, `reversibility`,
  `risk_level`, `governed_files`, `supersedes`, `superseded_by`, and
  `verification_status` when the decision should guide future implementation.
- Use `[[wikilinks]]` for notes inside the same vault. Use normal Markdown links
  only for external URLs.
- Use callouts sparingly:
  - `> [!summary]` for accepted outcome or decision summary
  - `> [!warning]` for unresolved risk, migration hazard, or compatibility concern
  - `> [!danger]` for code/note conflict or security-sensitive blocker
  - `> [!success]` for verified acceptance evidence
- Do not embed raw logs, secrets, or large generated reports in vault notes. Link to
  safe artifacts or summarize bounded evidence.

## Architecture index Base

Create an Obsidian `.base` dashboard when the user asks for a vault-side index,
decision dashboard, architecture ledger, or durable knowledge-base overview.

The deterministic index targets `obsidian.architecture_index_path`; default:
`architecture-index.base` under `allowed_root`.
Point `architecture_index_path` at an architect-coder-owned generated `.base`
file, not a hand-authored dashboard unless the user explicitly accepts
replacement.

Workflow:

1. Run `scripts/obsidian_preview_architecture_index.py --config <config>`.
2. Show the relative path, whether the file exists, `write_mode`,
   `content_sha256`, and preview.
3. Write only after authority is clear:
   `scripts/obsidian_write_architecture_index.py --config <config> --confirm-preview --expected-relative-path <path> --expected-sha256 <hash> --expected-write-mode <mode>`.
4. If the `.base` file already exists, also pass `--allow-replace` only after the
   replacement preview is accepted.

The generated Base filters to Markdown files under `allowed_root` tagged
`architect-coder`, then provides table views for all architecture records, ADRs,
acceptance notes, and worklogs. The ADR view exposes high-value decision metadata
such as scope, risk, reversibility, governed files, and verification status.

## Canvas maps

Create `.canvas` files only when the user asks for a visual architecture map or
there is clear value over Mermaid or prose. Validate every Canvas as JSON:

- top-level object with `nodes` and `edges` arrays
- unique 16-character lowercase hex IDs across nodes and edges
- every edge references existing node IDs
- node `type` is one of `text`, `file`, `link`, or `group`
- required fields are present for each node type

Use preview/hash binding before writing Canvas files, following the same safety
model as notes and Bases.

Do not add Canvas generation scripts until repeated real project workflows need
repeatable Canvas generation.
